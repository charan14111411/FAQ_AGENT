# Varsapradaya FAQ Chatbot — Comprehensive Deep Dive

This document provides a full, step-by-step explanation of exactly how the Varsapradaya FAQ agent works behind the scenes. It covers the flow, memory, database storage, and artificial intelligence routing.

---

## 1. The Core Architecture: What is LangGraph?

Traditional chatbots use complex `if/else` statements in a single file to figure out what the user wants. As chatbots grow, this becomes impossible to manage.

We built this agent using **LangGraph**, which treats the conversation as a **State Machine** (or a "Graph").
- **Nodes**: Think of nodes as "Rooms". There is a room for collecting the name, a room for collecting the email, and a room for answering FAQs.
- **State**: The `ChatState` is a backpack the user carries between rooms. It holds their name, phone number, email, and the conversation history.
- **Edges (Routing)**: The rules that dictate how a user moves from one room to the next.

### Why use a Checkpointer? (Memory)
Normally, when a user sends a message via an API (`POST /api/chat`), the server immediately forgets about them as soon as the request is done. 

The **Checkpointer** is LangGraph's long-term memory. When you send a message, you attach a `thread_id` (a unique ID generated by your frontend). 
- **Saving**: Every time the bot replies, the Checkpointer takes a "snapshot" of the user's backpack (`ChatState`) and saves it to the database (or MemorySaver during dev).
- **Loading**: When the user sends their next message with that same `thread_id`, the Checkpointer pulls the backpack out of storage so the bot remembers everything they've talked about.

## 2. Step-by-Step Flow Example

Let's walk through an example where a new user named "John" chats with the bot.

### Step 1: Greeting
- **Frontend sends:** `{"thread_id": "user-123", "message": "hi"}`
- **Graph Routing**: Because this is a new `thread_id`, the bot puts John in the `start` node.
- **Action**: The bot replies: *"Welcome! What's your full name?"*
- **Checkpointer**: Saves state `{"step": "await_name"}`.

### Step 2: Collecting Name
- **Frontend sends:** `{"thread_id": "user-123", "message": "John Doe"}`
- **Graph Routing**: The Checkpointer sees the last step was `await_name`. John is placed in the `collect_name_node`.
- **Action**: The bot runs validation. Does it contain letters? Yes. It saves `"John Doe"` in the backpack.
- **Bot Replies**: *"Nice to meet you, John! What's your phone number?"*
- **Checkpointer**: Saves state `{"step": "await_phone", "name": "John Doe"}`.

### Step 3: Collecting Phone & Email
- John provides his phone (or skips it). He then provides his email.
- Once the email is validated, a critical transition happens: **Database Interaction**.
- The bot checks the **`users` table** in PostgreSQL. If John's email exists, it loads his past preferences. Because he is new, it creates a new row in the `users` table.

### Step 4: Category Selection
- **Bot Replies**: *"What is your role? Grower, Corporate, Investor, or Agritech?"*
- **Frontend sends:** `{"thread_id": "user-123", "message": "I am a coffee farmer"}`
- **Action**: The bot uses a fast, lightweight LLM to classify "coffee farmer" into the `"grower"` category.
- **Database Interaction**: The bot creates a new row in the **`sessions` table**, linking John's `user_id` to the "grower" category.
- **Checkpointer**: Saves state `{"step": "chatting", "category": "grower"}`.

## 3. The RAG Engine (How it answers questions)

Once the user reaches the `chatting` step, they are fully onboarded. Every message they send now triggers the **RAG (Retrieval-Augmented Generation)** flow.

Let's say John asks: *"What spacing should I use for coffee plants?"*

1. **Save User Message**: The bot writes John's question to the **`messages` table** in PostgreSQL so we have a permanent record.
2. **Retrieve Context (Vector Search)**:
   - The bot converts John's text into numerical coordinates (an **Embedding**) using a local SentenceTransformer model.
   - It goes to the **`faq_embeddings` table** (which uses the `pgvector` extension) and performs a mathematical similarity search to find the 3 closest matching FAQs from your knowledge base.
3. **Build the Prompt**:
   - The bot loads the "Grower Persona" (e.g., *"Speak like a trusted agricultural advisor"*).
   - It pulls the last 10 messages from the `messages` table for conversational context.
   - It injects the 3 relevant FAQs it just found.
   - **Guardrails**: It adds strict instructions: *"Do not answer math questions or general knowledge. Only answer Varsapradaya-related questions."*
4. **Generate Answer**: The LLM (e.g., Groq/Llama3) reads the prompt and generates a factual response based *only* on the FAQ context provided.
5. **Save & Log**: The bot's answer is saved to the **`messages` table**. The latency, token usage, and AI model used are saved to the **`logs` table** for analytics.

## 4. The PostgreSQL Database Structure

Here is how the data is persistently stored outside of the active chat.

| Table Name | What it stores | When is it used? |
| :--- | :--- | :--- |
| **`users`** | Name, Email, Phone | Created once during the `await_email` step. Used to identify returning users across multiple days. |
| **`sessions`** | `user_id`, `category`, `status` | Created when the user selects a role (e.g., "Grower"). If the user types "switch to Investor", the old session is closed and a new one is created. |
| **`messages`** | `session_id`, `role` (user/assistant), `content` | Every time a user asks a question and the bot answers, both are saved here. Used to give the AI memory of the past 10 messages. |
| **`logs`** | Events, Errors, Metrics | Used for analytics. Tracks when users are created, when errors occur, and how many tokens the AI is using per response. |
| **`faq_embeddings`** | `question`, `answer`, `embedding` (vector) | Used by the RAG system to find the right factual answers to the user's specific question. |

## 5. Summary of Technologies Used

- **FastAPI**: Handles the web server and the `/api/chat` endpoint.
- **LangGraph**: Manages the State Machine, Checkpointing (memory), and routing users between onboarding and chatting.
- **PostgreSQL**: The relational database for permanent storage.
- **PGVector**: A PostgreSQL extension that allows the database to store and mathematically compare AI embeddings (used for RAG).
- **Groq/OpenAI**: The Large Language Models (LLMs) used to classify user intents and generate conversational answers.
- **SentenceTransformers**: A local, offline AI model that converts text into vector coordinates instantly without relying on paid APIs.
